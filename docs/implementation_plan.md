# Пошаговый план реализации

## Этап 0. Решения до начала разработки

1. Уточнить режим синхронизации:
   - только LAN;
   - LAN + собственный relay;
   - P2P с публичным rendezvous relay.
2. Выбрать renderer stack для форматов.
3. Определить политику DRM: поддерживается ли только non-DRM контент.
4. Определить платную/бесплатную модель, если нужны App Store / Google Play.

Проверка: утвержден ADR с транспортом, форматами, безопасностью и UX pairing.

## Этап 1. MVP библиотеки

1. Создать Flutter shell.
2. Реализовать локальный manifest/SQLite.
3. Добавить импорт файла.
4. Считать sha256, формат, размер.
5. Копировать книгу в локальное хранилище.
6. Показывать список книг, прогресс, статус скачанности.

Проверки:

- импорт одной и той же книги не создает дубль;
- книга сохраняется после перезапуска;
- статус `скачана` корректен после удаления локального файла;
- библиотека работает без сети.

## Этап 2. Reader MVP

1. TXT-reader как быстрый прототип.
2. EPUB/PDF reader через выбранный SDK.
3. Нормализация locator для каждого формата.
4. Автосохранение прогресса.
5. Закладки.

Проверки:

- закрыть книгу на 37%, открыть — позиция восстановлена;
- изменить шрифт/размер — locator остается валидным;
- закладка ведет в правильное место;
- большой файл не блокирует UI.

## Этап 3. Account и pairing

1. Создать локальную identity/account keypair.
2. Экран «Добавить устройство» с QR/одноразовым кодом.
3. Новое устройство получает manifest metadata.
4. Пользователь выбирает книги для скачивания.

Проверки:

- неподписанное устройство не может подключиться;
- код pairing истекает;
- новое устройство видит всю библиотеку, но файлы скачивает выборочно.

## Этап 4. Metadata sync

1. Sync envelope schema.
2. Обмен manifest между устройствами.
3. Merge rules.
4. Журнал изменений.
5. Conflict tests.

Проверки:

- offline-изменения на двух устройствах сливаются;
- прогресс выбирается по версии/времени;
- закладки не теряются;
- повторная доставка envelope идемпотентна.

## Этап 5. Book file sync

1. Chunking: 1–4 MB chunks.
2. Hash per chunk + sha256 всего файла.
3. Возобновление загрузки.
4. Шифрование chunk payload.
5. Выбор источника: любое онлайн-устройство, где есть файл.

Проверки:

- оборвать сеть на 60%, затем продолжить;
- испорченный chunk отвергается;
- итоговый sha256 совпадает;
- relay не пишет chunk на диск.

## Этап 6. Desktop/mobile hardening

1. Background sync с ограничениями iOS/Android.
2. Desktop tray/daemon.
3. Energy/battery policy.
4. Локальные уведомления об окончании sync.

Проверки:

- sync не жрет батарею;
- iOS не обещает фоновую передачу, если система ее не разрешила;
- desktop продолжает seed книг, пока приложение открыто или запущен helper.

## Этап 7. Production QA

1. Unit tests.
2. Golden UI tests.
3. Integration tests с двумя/тремя устройствами.
4. Security review.
5. Store compliance review.
6. Accessibility.

Проверки:

- тестовая матрица платформ закрыта;
- fuzz-тесты parser/locator;
- dependency/license audit;
- ручное тестирование импорта всех поддерживаемых форматов.
